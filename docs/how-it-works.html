02 The 5 stages, expanded

Click any stage to see what's actually happening, what tools are involved, and what can fail. The whole pipeline runs sequentially in roughly 5 minutes.

01

Trigger

The cloud server wakes up at 7:00 AM EDT

~1s

What happens

  • GitHub's cron scheduler fires at 11:00 UTC
  • A fresh Ubuntu virtual machine spins up
  • Latest code is checked out from main branch
  • Node.js, yt-dlp, Claude CLI install

Tools used

GitHub Actions cron: '0 11 * * *' ubuntu-latest Node 22

If it fails

  • GitHub sends you an email automatically
  • Re-run from Actions tab manually
02

Collect videos

Visit each channel, find yesterday's uploads, fetch metadata

~2 min

What happens

  • Read your channels.txt list
  • For each channel: ask YouTube for last 10 video metadata
  • Filter to videos uploaded yesterday (in your timezone)
  • Download captions/transcripts when available
  • Save raw data as JSON to a temp file

Tools used

yt-dlp YouTube cookies collect.js

Why cookies?

  • Cloud servers get blocked by YouTube
  • Your logged-in cookies bypass the block
  • Stored encrypted as GitHub Secret
03

Summarize with AI

Claude reads each video and writes a structured summary

~1.5 min

What happens

  • Claude CLI starts in headless mode
  • Reads the raw JSON file from step 2
  • For each video: writes 핵심요약 + 타임라인 + 인사이트
  • Groups summaries under red channel headers
  • Saves as a markdown file in tmp/

Tools used

Claude Code CLI --print mode claude-haiku-4-5 OAuth token

Why Haiku?

  • 2-3× faster than Sonnet
  • Uses less of your subscription quota
  • Quality is great for short summaries
04

Quality check

Validate every summary follows the right format

~1s

What happens

  • Read the summaries markdown file
  • Check every video has 핵심 요약 + 한 줄 인사이트
  • Verify timestamp format (HH:MM:SS)
  • Check at least one channel section exists
  • Auto-fix small format issues; warn on bigger ones

Tools used

review.js regex validators pure Node.js

Why this exists

  • AI sometimes drops a section header
  • Catches issues before publishing
  • Prevents broken pages in Notion
05

Publish + commit

Create Notion page, save backup to GitHub repo

~5s

What happens

  • Convert markdown to Notion blocks (h1/h2/h3, lists, quotes)
  • Create child page under "YouTube Summary" parent
  • POST blocks via Notion API (batched 100 at a time)
  • Apply red color to channel headers
  • Save same content to output/YYYY-MM-DD.md
  • Commit and push to GitHub for archive

Tools used

publish.js Notion API v1 git commit

Result

  • Notion page visible at 7:05 AM
  • You wake up, open Notion, read

05 When things go wrong

Every step has a failure mode. The system is designed to fail loudly and recover gracefully — most issues self-resolve on the next run.

YouTube blocks the request

YouTube's bot detection sometimes refuses to give metadata to cloud IPs, even with cookies. Your cookie file expired, perhaps.

Recovery: Re-export YouTube cookies (~60 days), update GitHub secret. Pipeline picks up next run.

Claude rate limit hit

If you used Claude heavily today, the workflow shares your 5-hour rolling quota. The pipeline returns "You've hit your limit · resets at X PM UTC".

Recovery: Auto-retry next morning with fresh quota. Or wait for reset and trigger manually.

Git push race condition

If GitHub's repo updated between checkout and push, the final commit step rejects with "fetch first".

Recovery: Built-in retry — pipeline pulls latest, rebases, pushes again. Up to 3 retries.

Notion permission revoked

If you removed the integration from your parent Notion page, the publish step returns 404.

Recovery: Re-add integration in Notion (·· menu → Connections). Backup still saves to GitHub.

Zero videos uploaded

Some days no channels post anything. The pipeline still runs successfully but produces an empty digest with a friendly message.

Recovery: Not a failure — just a quiet day. Empty Notion page is created with "수집된 영상이 없습니다" notice.

Format spec drift

You edited config/format.md but Claude wrote summaries in old format. Validation flags it as a warning.

Recovery: Pipeline still publishes — warnings don't block. Future runs follow new format automatically.

06 The 24-hour cycle

Once a day, a few minutes of work. The rest of the time the system is dormant.

One trigger. Five minutes of work. 23h 55min of standby.

The system sleeps until 7 AM EDT. Then it runs the full pipeline (collect → summarize → review → publish), commits the result, and goes back to sleep until tomorrow.
