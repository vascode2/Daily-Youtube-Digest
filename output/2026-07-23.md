## [The Unreasonable Effectiveness of Separating the Task from the Model — Maxime Rivest & Isaac Miller](https://www.youtube.com/watch?v=GgLQ02aO-hs)

**한 줄 인사이트**
💡 모델의 세부 구현과 프롬프트를 작업의 입출력(시그니처)에서 완전히 분리할 때, 변화하는 LLM 환경에 유연하게 대응하고 스스로 개선되는 AI 시스템을 구축할 수 있습니다.

**핵심 요약**
DSPy 핵심 기여자인 Maxime Rivest와 리드 메인테이너 Isaac Miller는 특정 프롬프트 템플릿이나 API 종속성에서 벗어난 AI 엔지니어링의 필요성을 역설합니다. 이들은 작업의 입력과 출력만을 정의하는 '시그니처' 개념을 통해 구현체와 작업을 분리함으로써 얻는 유연성을 설명합니다. 최종적으로 DSPy 3.5의 유용성을 짚어보고, 모델이 스스로 코드를 작성하고 사용자 상호작용으로 학습하는 DSPy 4.0의 청사진을 제시합니다.

1. **작업과 모델의 분리를 실현하는 시그니처(Signature)**
   - 모델의 기능에 구애받지 않고 작업의 입력과 출력만을 엄격하게 정의함으로써 워크플로우의 유연성을 극대화합니다.
   - 이를 통해 개발자는 전체 시스템의 흐름을 깨뜨리지 않고도 모델, 가중치, 프롬프트 템플릿, 출력 형식을 자유롭게 실험할 수 있습니다.

2. **DSPy 3.5의 성과와 프레임워크의 유용성**
   - 도구(Tools), 검색 증강 생성(RAG), 스킬이 복잡하게 얽힌 현대 AI 환경에서 시그니처 기반 설계는 시스템의 안정성을 보장합니다.
   - 프롬프트 엔지니어링의 수동적인 튜닝 작업을 자동화하고, AI 엔지니어링의 추상화 수준을 한 단계 높여줍니다.

3. **DSPy 4.0이 제시하는 자가 개선형 AI의 미래**
   - 차세대 버전인 DSPy 4.0에서는 모델이 시그니처 하위 단계에서 스스로 코드를 작성할 수 있는 기능을 지원할 예정입니다.
   - 프로그램이 시그니처의 입출력 규칙을 준수하면서도, 실제 사용자들과의 상호작용을 통해 직접 학습하고 성능을 개선해 나갑니다.

## [Notion's Token Town — Sarah Sachs, Notion](https://www.youtube.com/watch?v=-I5W5QVAT8E)

**한 줄 인사이트**
💡 AI 모델 공급업체를 경쟁자로 취급하고 모델에 종속되지 않는 독립성을 유지해야만 급격한 토큰 비용 상승 속에서 제품 경쟁력을 확보할 수 있습니다.

**핵심 요약**
Notion의 AI 엔지니어링 리드인 Sarah Sachs는 매달 반복되는 AI 모델의 가격 인상과 토큰 소비 함정 속에서 기업이 살아남는 전략을 제시합니다. 그녀는 공급업체에 종속되지 않고 자체적인 오케스트레이션과 모델 불가지론(Model Agnostic) 접근법을 취해야만 비즈니스 마진을 지킬 수 있다고 주장합니다. 이를 위해 Notion이 실제로 구축한 자동 라우팅 시스템과 오픈 웨이트 모델 활용법을 구체적인 사례와 함께 설명합니다.

1. **공급업체를 경쟁자로 인식하는 모델 독립성 확보**
   - 자체 제품을 판매하는 AI 연구소로부터 토큰을 구매하는 것은 마진에 마진을 더하는 방어 불가능한 구조이므로, 특정 제공업체에 종속되지 않는 'AI 스위스' 전략이 필수적입니다.
2. **효율적인 자동 라우팅 시스템 구축**
   - Notion은 전체 트래픽의 약 75%를 자동으로 라우팅하여 백엔드에서 모델 제공업체를 유연하게 전환하는 오토 모델(Auto Model)을 운영하고 있습니다.
   - 단순히 토큰당 비용이 아닌 기능당 성능과 속도(cost per capability per second)를 기준으로 트래픽을 분배하여 불필요한 비용 낭비를 막아야 합니다.
3. **오픈 웨이트 모델과 CPU의 전략적 활용**
   - 중간 수준의 작업에는 오픈 웨이트(Open Weight) 모델을 적극적으로 도입하고, CSV 파일을 PDF로 변환하는 등의 단순 작업에는 GPU 대신 CPU를 활용해 비용을 극적으로 절감합니다.
4. **제품 경쟁력과 에이전트 오케스트레이션**
   - 토큰 경제성 싸움에서 벗어나 데이터 플라이휠과 오케스트레이션 등 제품 자체의 가치에 집중해야 하며, Notion 에이전트가 작업을 분석하고 PR을 생성하는 등의 실질적인 워크플로우를 구현해야 합니다.

## [Harness Engineering is not Enough: Why Software Factories Fail — Dex Horthy, HumanLayer](https://www.youtube.com/watch?v=Ib5GBkD555M)

**한 줄 인사이트**
💡 코딩 AI 모델은 단순히 테스트 통과에만 최적화되어 있어 코드 유지보수성과 아키텍처 품질을 스스로 관리하지 못하므로, 인간 개발자의 사전 설계와 코드 리뷰가 여전히 필수적입니다.

**핵심 요약**
HumanLayer의 Dex Horthy는 2025년 7월 사람이 코드를 읽지 않는 에이전트 소프트웨어 공장을 운영하다가 시스템이 완전히 붕괴한 실제 경험을 공유합니다. 그는 코딩 AI 모델이 테스트 통과라는 단기적 보상에만 길들여져 있어 장기적인 코드 아키텍처와 유지보수성을 망가뜨린다고 주장합니다. 이를 해결하기 위해 개발자가 다시 개입하여 제품 설계, 시스템 아키텍처, 타입 정의 등 사전 계획 단계에 집중하는 '불을 다시 켜는(turn the lights back on)' 접근법을 제시합니다.

1. **에이전트 소프트웨어 공장의 실패 사례**
   - Dex Horthy는 2025년 7월 아무도 코드를 읽지 않는 완전 자동화된 에이전트 소프트웨어 공장 실험을 진행했으나, 프롬프팅으로 해결할 수 없는 장애가 발생하며 실패했습니다.
   - 사이트가 다운되고 사용자들이 분노하는 상황에서, 그는 지난 3개월 동안 한 번도 읽지 않았던 코드베이스를 직접 뒤지며 복구 작업을 해야 했습니다.

2. **코딩 모델의 한계와 테스트 통과 편향**
   - 현재 코딩 AI 모델들은 기존 테스트를 깨뜨리지 않고 새로운 테스트를 통과하는지 여부로만 강화학습(RL)을 받기 때문에 코드의 유지보수성을 고려하지 못합니다.
   - 이러한 학습 방식은 단기적으로는 테스트를 통과하지만, 장기적으로는 나쁜 아키텍처가 누적되어 몇 달 뒤에 엄청난 비용을 치르게 만드는 원인이 됩니다.

3. **Claude Code의 성공 요인과 벤치마크의 한계**
   - Claude Code가 단기간에 수억 달러 가치로 성장한 이유는 단순히 읽기/쓰기 명령어가 우수해서가 아니라, 실제 배포되는 하네스(Harness) 환경에 맞춰 직접 훈련된 최초의 모델이기 때문입니다.
   - 하지만 유지보수성은 녹색 테스트 패스처럼 쉽게 검증할 수 없으며, 모델이 좋은 코드의 기준을 스스로 이해하고 작성하는 데는 여전히 한계가 존재합니다.

4. **'불을 다시 켜는' 사전 계획 설계법**
   - AI 시대에도 개발자는 코드 읽기를 멈추지 않아야 하며, 제품 검토, 시스템 아키텍처 설계, 타입 및 호출 그래프 정의 등 사전 설계에 30분을 투자해야 합니다.
   - 이러한 사전 조율은 이후 수 시간의 코드 리뷰와 디버깅 시간을 절약해 주며, 에이전트가 생성한 스파게티 코드를 정리하는 대신 즐거운 PR 검토 환경을 만들어 줍니다.

3. **지속 가능한 개발과 성장을 위한 대안**
   - Linear와 같은 효율적인 협업 도구를 활용하여 업무 프로세스를 체계화하고 불필요한 오버워크를 줄여야 합니다.
