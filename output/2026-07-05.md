## [삼성은 분리하고 화웨이는 접고… 스마트폰 칩 전쟁 속 극과 극의 솔루션](https://www.youtube.com/watch?v=HPLOGiIXyjg)

**한 줄 인사이트**
💡 모바일 칩의 발열 문제를 해결하기 위해 삼성은 메모리를 측면으로 분리해 열 배출구를 넓히고, 화웨이는 회로를 수직으로 접어 쌓는 완전히 상반된 패키징 전략을 취하고 있습니다.

**핵심 요약**
스마트폰을 장시간 사용할 때 발생하는 발열과 성능 저하(스로틀링) 문제를 해결하기 위해 글로벌 반도체 기업들이 패키징 기술 혁신에 사활을 걸고 있습니다. 안될공학은 삼성이 메모리 배치를 변경해 방열 성능을 극대화하는 방식과 화웨이가 회로를 수직으로 접어 쌓는 적층 기술을 비교 분석합니다. 나아가 애플과 퀄컴의 최신 기술 동향까지 짚어보며, 미세 공정의 한계를 극복하려는 빅테크 기업들의 차세대 반도체 패키징 트렌드를 명확히 설명합니다.

1. **삼성의 발열 해결을 위한 메모리 분리 전략**
   - 삼성은 기존에 칩 상단에 메모리를 쌓던 방식에서 벗어나 메모리를 측면으로 배치함으로써 열이 빠져나갈 수 있는 물리적 경로를 넓히는 설계를 도입했습니다.
   - 엑시노스 2400의 팬아웃(Fan-Out) 패키징과 차기 엑시노스 2600의 히트 패스 블록 적용, 그리고 엑시노스 2700을 둘러싼 루머를 통해 삼성의 방열 기술 진화 과정을 보여줍니다.

2. **화웨이의 수직 적층 및 로직 폴딩 기술**
   - 화웨이는 칩 내부의 회로를 수직으로 접어서 쌓아 올리는 로직 폴딩(Logic Folding) 기술과 하이브리드 본딩에 관한 논문을 발표하며 완전히 다른 길을 제시했습니다.
   - 이는 미세 공정 장비 수급이 어려운 상황에서 칩의 구조적 설계를 3차원으로 확장하려는 시도로, 과장된 소문 속에서 실제 기술적 변화의 실체를 파악하는 것이 중요합니다.

3. **애플과 퀄컴의 패키징 기술 대응 현황**
   - 모바일 칩 시장의 강자인 애플과 퀄컴 역시 발열과 성능 저하 문제를 해결하기 위해 각자 차별화된 패키징 솔루션과 기술 트렌드를 추구하고 있습니다.
   - 스마트폰의 성능이 상향 평준화됨에 따라 단순한 미세 공정 경쟁을 넘어 효율적인 열 배출과 전력 효율을 동시에 달성하는 패키징 고도화 경쟁으로 패러다임이 전환되고 있습니다.

4. **차세대 모바일 칩의 진정한 완성 조건**
   - 단순히 트랜지스터의 크기를 줄여 미세화하는 것만으로는 더 이상 고성능 모바일 칩의 발열과 전력 소모 문제를 완벽히 해결할 수 없습니다.
   - 열 설계 전력을 효율적으로 관리하고 칩 내부의 열을 어떻게 외부로 신속하게 방출할 것인가가 향후 스마트폰 AP 전쟁의 성패를 가를 핵심 열쇠입니다.

## [The Missing Layer After Launch - Raphael Kalandadze, Wandero AI](https://www.youtube.com/watch?v=kZsf_Sfm7RU)

**한 줄 인사이트**
💡 에이전트 시스템의 실패는 에러 로그가 아닌 대화 속에서 발생하므로, 모니터링과 테스트를 수행하는 운영 레이어 역시 에이전트 팀으로 자동화해야 합니다.

**핵심 요약**
Wandero AI의 공동 창업자이자 CTO인 Raphael Kalandadze는 기존 소프트웨어와 달리 에이전트 제품의 실패는 시스템 다운이 아닌 대화의 맥락에서 발생한다고 주장합니다. 그는 이러한 비결정론적 문제를 해결하기 위해 모니터링, 테스트, PR 리뷰를 스스로 수행하는 '운영 에이전트 팀'의 필요성과 구축 방법을 제시합니다.

1. **대화 맥락 중심의 실패 모니터링**
   - 에이전트의 실패는 에러 로그가 아닌 대화 속 제약 조건 누락이나 잘못된 답변으로 나타나므로, 운영 에이전트가 수천 개의 실시간 세션을 직접 모니터링하며 평가해야 합니다.
2. **비결정론적 시스템을 위한 에이전트 기반 테스트**
   - 전통적인 CI 환경의 통과 여부는 에이전트의 무작위적 행동을 보장하지 못하기 때문에, 테스트 에이전트가 직접 시나리오를 작성하고 실행하며 안정성을 검증합니다.
3. **코드 변경 사항에 대한 에이전트의 PR 리뷰**
   - 인간 또는 에이전트가 작성한 모든 Pull Request(PR)를 에이전트가 검토하여, 변경 사항이 근본적인 원인 해결인지 아니면 임시방편적인 증상 완화인지 판별합니다.
4. **인간과 에이전트의 협업 경계 설정**
   - 대량의 모니터링, 테스트, 초안 작성 등의 고된 작업은 에이전트가 전담하되, 최종 코드 병합(Merge)과 승인 권한은 인간 엔지니어가 유지하여 안전성을 확보합니다.

## [Continual Learning for AI Agents: From Failures to Durable Improvements - Soheil Feizi, RELAI](https://www.youtube.com/watch?v=2IxD9OB3XuQ)

**한 줄 인사이트**
💡 AI 에이전트가 실서비스에서 겪는 실패와 피드백을 회귀 오류 없이 지속적인 성능 향상으로 전환하려면 검증 가능한 지속적 학습(Verifiable Continual Learning) 프레임워크가 필수적입니다.

**핵심 요약**
RELAI의 창립자이자 메릴랜드 대학교 부교수인 소헤일 페이지(Soheil Feizi) 박사는 정적 벤치마크가 포착하지 못하는 실제 운영 환경의 에이전트 실패 문제를 해결하기 위한 새로운 접근법을 제시합니다. 그는 에이전트가 기존 능력을 망가뜨리지 않으면서 실패 경험으로부터 안전하게 학습할 수 있도록 돕는 '검증 가능한 지속적 학습'의 핵심 요구사항과 아키텍처를 설명합니다. 이를 통해 프롬프트 최적화와 메모리 통합 등 기존 기술의 한계를 짚어보고, 온라인 회귀 제어가 가능한 평생 학습 루프의 필요성을 역설합니다.

1. **실패 사례의 재현 가능한 학습 환경 전환**
   - 실서비스에서 발생하는 에이전트의 실패 궤적(Trace)과 사용자 피드백을 테스트 가능한 재현 환경으로 변환하는 것이 지속적 학습의 첫걸음입니다.
   - 이를 통해 정적 벤치마크를 넘어 실제 실패 시나리오를 기반으로 에이전트의 취약점을 정밀하게 진단하고 개선할 수 있습니다.

2. **업데이트 과정에서의 이전 기능 보존**
   - 새로운 실패를 학습하고 에이전트를 업데이트할 때, 기존에 잘 작동하던 기능이 망가지는 회귀(Regression) 현상을 방지해야 합니다.
   - 안전한 성능 향상을 위해서는 업데이트 전후의 기능을 엄격하게 비교 검증할 수 있는 회귀 인식형(Regression-aware) 평가 체계가 필수적입니다.

3. **에이전트 스택 내 적절한 레이어로의 복구 경로 지정**
   - 에이전트의 오류를 해결하기 위해 프롬프트 최적화, 메모리 통합, 코드 하네스 수정 등 에이전트 아키텍처의 어떤 레이어를 수정할지 정확히 라우팅해야 합니다.
   - 각 레이어의 특성에 맞는 복구 메커니즘을 적용해야만 시스템 전체의 복잡성을 낮추고 효율적인 수리가 가능해집니다.

4. **지속 가능한 온라인 회귀 제어 루프 구축**
   - 실시간으로 학습 루프를 가동하기 위해서는 연산 및 시간 측면에서 지속적으로 실행 가능할 만큼 가볍고 효율적인 시스템을 설계해야 합니다.
   - 궁극적으로는 전체 에이전트 수명 주기 동안 안전성을 보장하는 총체적이고 검증 가능한 평생 학습(Lifelong Learning) 루프를 완성하는 것이 목표입니다.

## [MCP Apps: Primitives, discovery, and the Future of Software - Pietro Zullo, Manufact, Inc](https://www.youtube.com/watch?v=sAOBXCDiDOs)

**한 줄 인사이트**
💡 MCP 앱은 단순한 서버 확장을 넘어 모델과 UI가 실시간으로 상태를 공유하는 양방향 상호작용 레이어로서 에이전트 시대의 새로운 소프트웨어 표준이 될 것입니다.

**핵심 요약**
Manufact의 공동 창업자인 Pietro Zullo는 AI 에이전트의 대중화에 발맞춰 기존 소프트웨어 기업들이 생존하기 위해 왜 MCP(Model Context Protocol) 앱을 도입해야 하는지 설명합니다. 그는 MCP 앱의 고유한 아키텍처와 핵심 프리미티브를 분석하고, 실제 작동하는 데모와 배포 방식을 통해 개발자들이 이 새로운 생태계에 참여할 수 있는 구체적인 경로를 제시합니다.

1. **MCP 앱의 정의와 아키텍처**
   - MCP 앱은 단순한 UI가 결합된 서버가 아니라, `ui://` 리소스를 통해 선언되고 호스트의 샌드박스 iframe 내에서 렌더링되는 완전한 양방향 상태 보존형 상호작용 레이어입니다.
   - JSON-RPC-over-postMessage 전송 방식을 통해 호스트와 통신하며, AI 모델과 사용자 인터페이스가 실시간으로 컨텍스트를 공유하도록 설계되었습니다.

2. **작동을 가능하게 하는 핵심 프리미티브**
   - `ui/update-model-context` 프로토콜을 사용하면 사용자의 추가 메시지 없이도 앱이 모델의 컨텍스트 창에 실시간 상태를 직접 주입할 수 있습니다.
   - `ui/message`를 통해 앱이 대화에 능동적으로 참여하고, 모델은 앱에 등록된 도구(App Tools)를 호출하여 상호작용을 완성합니다.

3. **배포, 발견 및 시장의 변화**
   - 개발자는 전용 스토어를 통해 MCP 앱을 제출하고 배포할 수 있으며, 사용자는 다양한 호스트 환경에서 이를 손쉽게 검색하고 설치할 수 있습니다.
   - 인간이 UI를 통해 사용하는 모든 기존 소프트웨어 제품은 에이전트 중심의 환경에서 소외되지 않기 위해 결국 MCP 앱 버전을 구축해야 할 것입니다.

4. **Manufact와 mcp-use 생태계**
   - Manufact는 PyPI와 npm을 통해 800만 회 이상 다운로드된 선도적인 MCP 개발 프레임워크인 `mcp-use`를 유지 관리하며 MCP 앱의 클라우드 인프라를 제공합니다.
   - 이들은 웹 개발에서 Next.js와 Vercel의 관계처럼, MCP 앱과 서버 개발 및 배포를 위한 수직 통합형 플랫폼 역할을 지향합니다.
