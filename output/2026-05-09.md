## [Why TTS Models Now Look Like LLMs — Samuel Humeau, Mistral](https://www.youtube.com/watch?v=3jGAU2sbAyY)

**한 줄 인사이트**
💡 음성 합성이 "오디오 토큰을 자기회귀로 생성하는 LLM"으로 수렴하면서, TTS도 LLM과 동일한 학습·추론 도구체인을 공유하는 단일 시스템 문제가 되고 있습니다.

**핵심 요약**
Mistral의 Sam Humeau가 자사 첫 오픈소스 TTS 모델을 공개하며, "왜 요즘 TTS가 LLM처럼 생겼는가"를 처음부터 설명하는 발표.

1. **TTS의 핵심 use case는 이제 voice agent**
   - 텍스트 LLM 위에 음성 모듈을 얹는 "speech wrapper" 구조가 표준이 됨. [[02:02](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=122)]
   - 이 구조에서는 **첫 오디오까지의 latency**가 사용자 체감을 좌우. 텍스트 생성이 끝나길 기다리지 않고 토큰 단위로 흘려보내야 함. [[02:31](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=151)]

2. **오디오를 "토큰 시퀀스"로 보는 패러다임**
   - 인간 음성의 정보량은 초당 **약 15비트** — MP3 200 kbps와 비교하면 압축 여지가 엄청남. [[11:09](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=669)]
   - 그래서 코덱이 80ms 프레임을 **초당 12프레임 정도의 토큰**으로 양자화 → LLM 입력 형태로 동일화. [[12:00](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=720)]
   - 결과: TTS도 텍스트 LLM과 같은 인코더-디코더 + 자기회귀 구조 사용 가능. [[09:09](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=549)]

3. **라이브 데모: 음성 클로닝과 다국어**
   - Paul이라는 실제 인물의 목소리를 **수 초 분량 샘플로 복제**해 합성. [[04:06](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=246)]
   - 같은 모델이 영어 화자의 목소리로 **프랑스어를 구사**하는 cross-lingual 합성 시연. [[06:24](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=384)]
   - 단일 GPU에서 첫 오디오까지 **17ms** 지연. [[16:14](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=974)]

4. **사회적 함의: 음성 신원 보호**
   - 누구든 짧은 샘플로 타인을 "사칭"할 수 있게 됨 → voice identity 보호가 시급한 과제로 부상. [[07:23](https://www.youtube.com/watch?v=3jGAU2sbAyY&t=443)]

## [Voice AI: when is the "Her" moment? — Neil Zeghidour, Gradium AI](https://www.youtube.com/watch?v=P_RI1kCkRbo)

**한 줄 인사이트**
💡 "Her 순간"의 병목은 모델 지능이 아니라 **풀듀플렉스 + 백채널링 + 200ms 이하 latency**라는 시스템 디자인 문제이며, 이것을 풀려면 STT/LLM/TTS 캐스케이드를 통합 음성 모델로 대체해야 한다는 주장.

**핵심 요약**
Kyutai에서 Moshi를 만들었던 Neil Zeghidour가 Gradium AI 창업 후 다시 무대에 올라, 영화 *Her* 수준의 음성 대화가 가능한 시점을 시스템 관점에서 진단.

1. **현재 음성 에이전트의 한계: "음성을 두른 텍스트 모델"**
   - 대부분의 voice agent는 텍스트 LLM에 STT/TTS를 양옆에 붙인 **glorified text model with a voice around** 구조. [[05:30](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=330)]
   - 캐스케이드(STT → LLM → TTS) 구조에서 응답까지의 latency가 본질적으로 높음 — 약 **200ms는 TTS 시작에만** 소비. [[06:22](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=382)]
   - 여기에 tool call이 끼면 latency가 더 예측 불가해짐. [[07:03](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=423)]

2. **풀듀플렉스 vs 하프듀플렉스**
   - 인간 대화는 풀듀플렉스 — 상대가 말하는 동안에도 "음", "어" 같은 **백채널링(backchanneling)**을 끊임없이 보냄. [[10:20](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=620)]
   - 캐스케이드 구조는 풀듀플렉스를 흉내내려 하면 즉시 깨짐 — 끼어들기·중첩 발화 처리 불가. [[10:10](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=610)]

3. **Moshi의 해법: 음성 입출력 통합 모델**
   - STT/LLM/TTS를 한 모델로 합쳐 **단계 수 자체를 줄여** latency 문제 해결. [[13:24](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=804)]
   - 라이브 데모에서 사용자가 말 도중에 끊어도 모델이 자연스럽게 백채널 + 화제 전환. [[11:07](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=667)]

4. **온디바이스 + 비용·프라이버시**
   - 거대 멀티모달 음성 모델은 **API 호출 1건당 손실**을 보고 운영되는 상황. [[15:58](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=958)]
   - Gradium의 차별점: 모델이 **스마트폰 CPU에서 로컬 실행** → API 비용 0, 프라이버시 보장. [[17:13](https://www.youtube.com/watch?v=P_RI1kCkRbo&t=1033)]

## [Give Your Chat Agent a Voice — Luke Harries, ElevenLabs](https://www.youtube.com/watch?v=DCZZ3AJKzuc)

**한 줄 인사이트**
💡 모든 챗 에이전트가 음성 에이전트로 업그레이드되는 흐름에서, 모델을 다시 짜는 게 아니라 **기존 텍스트 에이전트에 SDK 한 겹을 wrapping**해 음성 I/O만 외주화하는 접근이 가장 빠른 시장 진입 경로.

**핵심 요약**
ElevenLabs의 Luke Harries가 신제품 early preview를 공개. 이미 동작하는 텍스트 챗 에이전트에 음성을 얹는 방법을 데모.

1. **왜 챗을 음성으로 바꿔야 하는가**
   - 키보드 사용이 어렵거나 난독증인 사용자에게 **음성이 훨씬 자연스러운 매체**. [[01:13](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=73)]
   - "모든 챗 에이전트는 결국 voice agent로 업그레이드되어야 한다"는 ElevenLabs의 산업 예측. [[01:37](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=97)]

2. **고객 페인포인트: 처음부터 다시 만들기 싫음**
   - 기존에 LangGraph 등으로 챗 에이전트를 만들어둔 회사들이 "음성을 위해 처음부터 다시 짜라고?"라며 반발. [[02:37](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=157)]

3. **통합 코드 구조 — 단순한 프록시 패턴**
   - `client → voice engine → 기존 chat agent`로 호출을 그대로 프록시. [[03:53](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=233)]
   - 클라이언트 SDK 통합 후에는 **코딩 에이전트(Cursor 등)에게 코드베이스를 넘겨** 자동으로 voice 배포까지 가능. [[04:45](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=285)]

4. **도구 호출(tool call) 통합**
   - client-side / server-side tool 모두 wrapping된 에이전트로 프록시 가능 — 기존 비즈니스 로직 보존. [[07:27](https://www.youtube.com/watch?v=DCZZ3AJKzuc&t=447)]

## [Microsoft doesn't understand FPS](https://www.youtube.com/watch?v=znRHdCDw9tQ)

**한 줄 인사이트**
💡 같은 성능 데이터라도 FPS냐 ms(프레임타임)이냐에 따라 청자 반응이 갈린다는 점은, 엔지니어가 사용자·임원에게 지표를 제시할 때 단위 선택 자체가 커뮤니케이션 전략임을 보여줍니다.

**핵심 요약**
PrimeTime이 "윈도우 Run 다이얼로그가 11 fps로 떨어진다"고 묘사했다가 시청자들에게 "FPS를 그렇게 쓰면 안 된다"고 공격받은 사건을 풀어내는 라이브 잡담. 자신의 측정 사례와 시청자 반응을 비교하며 단위 선택 논쟁을 정리.

1. **사건 발단 — 11 fps라는 표현**
   - 발표자는 94ms 지연을 환산해 **"약 11 fps"** 로 표현. [[03:23](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=203)]
   - 핵심은 "체감 반응성"을 직관적으로 묘사하는 단위로 FPS를 쓴 것. [[03:44](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=224)]

2. **시청자 반응 — "FPS는 평균값으로만 쓰는 단어다"**
   - 게이밍 커뮤니티 일부의 통념: **FPS = average FPS만** 의미해야 함. [[06:44](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=404)]
   - 발표자는 이걸 가상의 "internet committee"가 강제한 규칙이라며 풍자. [[07:37](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=457)]

3. **반박 — ms와 fps는 같은 데이터의 다른 표현**
   - "in this house we measure with FPS" 농담으로 통념을 풍자. [[07:13](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=433)]
   - "fair FPS measure"라는 게 따로 있는 게 아니라, **목적에 맞는 단위**를 고르면 됨. [[08:17](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=497)]

4. **엔지니어링 함의**
   - 펌웨어/시스템 엔지니어에겐 ms가, UX/제품팀엔 fps가 직관적 — 같은 청중이라도 단위 선택으로 토론 결과가 달라질 수 있다는 메타 교훈. [[35:00](https://www.youtube.com/watch?v=znRHdCDw9tQ&t=2100)]

## [실리콘밸리 엘리트들 생각](https://www.youtube.com/watch?v=zmFPt79QKC8)

**한 줄 인사이트**
💡 a16z 마크 안드레센이 말하는 "방치의 실수(omission)가 실행의 실수(commission)보다 더 비싸다"는 사고방식은, 보수적으로 신기술 도입을 늦출 때 잃는 기회비용을 의식적으로 환산해야 함을 보여줍니다.

**핵심 요약**
$90B 운용 a16z의 공동창업자 **마크 안드레센(Mark Andreessen)** 인터뷰 요약. 실수의 종류, 위대한 창업자 조건, AI와 노동 시장 전망을 중심으로 정리.

1. **벤처 캐피털의 사고방식: '실수로부터 배우기'의 함정**
   - **실행의 실수(commission) vs 방치의 실수(omission)**: $10M을 잃는 것보다 Google 같은 회사를 놓쳐 $100B 기회비용을 잃는 게 훨씬 큼. 벤처는 후자를 더 경계해야 함. [[03:02](https://www.youtube.com/watch?v=zmFPt79QKC8&t=182)]
   - **반성(introspection)의 함정**: "Hot Stove" 비유처럼, 한 번 데인 경험으로 미래 기회를 회피하는 게 가장 큰 리스크. 1945년 이후 수십 년의 AI 겨울을 거친 투자자들이 지금 혁신을 못 보는 이유. [[01:41](https://www.youtube.com/watch?v=zmFPt79QKC8&t=101)]
   - a16z는 파트너 전체에 "anti-scalded(데이지 않은) 마인드"를 주입해 omission 리스크를 우선시하도록 훈련. [[03:53](https://www.youtube.com/watch?v=zmFPt79QKC8&t=233)]

2. **위대한 창업자의 3가지 조건**
   - **High IQ**: 대화 중 노트를 많이 적게 만드는지로 판별. [[04:26](https://www.youtube.com/watch?v=zmFPt79QKC8&t=266)]
   - **용기(Courage)**: 파트너 Ben Horowitz의 표현. 어떤 문제든 정면 돌파해서 끝까지 해내겠다는 결단력. [[05:01](https://www.youtube.com/watch?v=zmFPt79QKC8&t=301)]
   - **야망(Drive / Will to Power)**: 니체를 인용. 단순 문제 해결이 아니라 "내 것을 세우고 세상을 바꾸겠다"는 원초적 욕구. [[05:39](https://www.youtube.com/watch?v=zmFPt79QKC8&t=339)]

3. **창업자의 성장 배경 — 결핍형 vs 안정형**
   - 결핍·트라우마형 예시로 **스티브 잡스**(broken home 배경) 언급. [[07:04](https://www.youtube.com/watch?v=zmFPt79QKC8&t=424)]
   - 그러나 **마크 저커버그**, **빌 게이츠**처럼 안정적 가정에서도 10대부터 강력한 drive를 보인 사례가 많음. 결핍이 필수 조건은 아님. [[07:54](https://www.youtube.com/watch?v=zmFPt79QKC8&t=474)]

4. **AI와 노동 시장: "Lump of Labor" 오류**
   - 일자리 총량 고정이라는 통념은 **고전적 제로섬 경제학 오류(lump of labor fallacy)**라며 단호히 부정. [[16:25](https://www.youtube.com/watch?v=zmFPt79QKC8&t=985)]
   - 최근 빅테크 해고는 AI 때문이 아니라 **금리 인상 + 코로나 시기 25~50% 과잉 고용**의 후유증. AI는 명분일 뿐. [[18:52](https://www.youtube.com/watch?v=zmFPt79QKC8&t=1132)]
   - **소비자 잉여(Consumer Surplus)**: 인터넷·스마트폰처럼 가치의 99%는 사용자에게, 1%만 Apple·Google에 귀속. AI도 동일 패턴 예상. [[14:30](https://www.youtube.com/watch?v=zmFPt79QKC8&t=870)]

5. **미국의 경쟁력 — 리스크 테이킹 DNA**
   - 미국 사회는 "newcomer 환영 + 새로운 시도에 대한 동력"이 DNA에 박혀 있어 일론 머스크 같은 성취가 가능했음. [[09:02](https://www.youtube.com/watch?v=zmFPt79QKC8&t=542)]
