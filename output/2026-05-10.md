## [Two Roads to Durable Agents: Replay vs. Snapshot — Eric Allam, Trigger.dev](https://www.youtube.com/watch?v=svCnShDvgQg)

**한 줄 인사이트**
💡 장기 실행 에이전트의 내구성을 확보하려면 기존 리플레이 모델의 한계를 넘어, 컨텍스트 로그와 실행 스냅샷/복원 기능을 결합한 스테이트풀 컴퓨팅으로의 전환이 필수적이다.

**핵심 요약**
LLM 에이전트의 장기 실행 내구성을 확보하는 것은 현대 백엔드 인프라의 중요한 과제입니다. 기존의 리플레이(Replay) 모델은 한계가 있으며, 에이전트의 컨텍스트와 실행 상태를 분리하여 관리하는 새로운 접근 방식이 필요합니다.

*   **기존 백엔드 아키텍처와 Replay 모델의 한계**
    *   CGI (1993)부터 PHP, Ruby on Rails, Node.js, Serverless까지 지난 30년간 백엔드 인프라는 `shared nothing architecture` (컴퓨트 레이어는 stateless, 상태는 DB에 저장) 패러다임을 따랐다. [[2:47](https://www.youtube.com/watch?v=svCnShDvgQg&t=167)]
    *   복잡한 웹 애플리케이션에서 비동기 다단계 작업을 처리하기 위해 10~15년 전부터 `replay model` 기반의 워크플로우 및 내구성 있는 실행 엔진이 도입되었다. 이는 각 사이드 이펙트를 단계로 래핑하고 실행 후 캐시하여 실패 시 재시작해도 중복 실행을 방지한다. [[3:41](https://www.youtube.com/watch?v=svCnShDvgQg&t=221)]
    *   2023년 LLM 에이전트가 등장하면서, 모든 LLM 호출과 툴 호출이 리플레이 저널의 단계가 되어 로그가 빠르게 증가하고 시스템 한계에 도달할 수 있다. 에이전트는 트랜잭션이 아닌 세션처럼 작동하며, 장기 실행 에이전트의 지속 시간은 4~7개월마다 두 배로 증가하고 있어 리플레이 모델만으로는 한계가 있다. [[5:05](https://www.youtube.com/watch?v=svCnShDvgQg&t=305)], [[6:46](https://www.youtube.com/watch?v=svCnShDvgQg&t=406)]

*   **내구성 있는 에이전트를 위한 두 가지 핵심 요소**
    *   에이전트의 내구성은 크게 두 가지로 나뉜다: **컨텍스트 내구성(Context Durability)**과 **실행 내구성(Execution Durability)**. [[7:27](https://www.youtube.com/watch?v=svCnShDvgQg&t=447)]
    *   **컨텍스트 내구성**: LLM의 시스템 메시지, 사용자 메시지, 툴 호출, 툴 결과, 어시스턴트 응답 등 LLM 입출력에 대한 모든 컨텍스트를 포함한다. 이는 데이터베이스, 객체 스토리지, 분산 파일 시스템과 같은 기존 기술을 사용하여 append-only 로그 형태로 쉽게 저장하고 버전 간 내구성을 확보할 수 있다. [[8:07](https://www.youtube.com/watch?v=svCnShDvgQg&t=487)]
    *   **실행 내구성**: 에이전트가 GitHub 리포지토리 클론, 패키지 설치, 메모리 내 데이터셋, 개발 서버 실행 등 컴퓨트 레이어에서 수행하는 모든 상태를 의미한다. 이는 로그만으로는 내구성을 확보하기 어렵다. [[8:54](https://www.youtube.com/watch?v=svCnShDvgQg&t=534)]

*   **Snapshot 및 Restore를 통한 실행 내구성 확보**
    *   실행 상태를 로그에서 재구성하는 대신, `snapshot and restore` 방식을 사용해야 한다. 이는 머신 상태를 스냅샷으로 저장하고 종료한 후, 필요할 때 복원하여 중단된 지점부터 다시 시작할 수 있게 한다. 이는 턴(turn) 간 내구성을 제공하며 비용 효율적이다. [[9:33](https://www.youtube.com/watch?v=svCnShDvgQg&t=573)]
    *   이러한 스냅샷 및 복원 기능은 1966년 IBM 메인프레임의 체크포인트/복원 기능과 유사하며, 2011년 CRIU (Checkpoint/Restore In User-space) 개발로 프로세스 단위 스냅샷이 가능해졌다. [[11:26](https://www.youtube.com/watch?v=svCnShDvgQg&t=686)], [[11:51](https://www.youtube.com/watch?v=svCnShDvgQg&t=711)]
    *   Trigger.dev는 2024년에 Firecracker microVM을 사용하여 전체 머신 스냅샷 및 복원 기능을 구현했다. 이를 통해 512MB 기본 머신 스냅샷을 압축하여 14MB까지 줄였으며, 스냅샷은 1초 미만, 복원은 수백 밀리초 내에 완료된다. [[12:58](https://www.youtube.com/watch?v=svCnShDvgQg&t=778)], [[14:05](https://www.youtube.com/watch?v=svCnShDvgQg&t=845)], [[14:36](https://www.youtube.com/watch?v=svCnShDvgQg&t=876)]
    *   Trigger.dev는 이 기술을 `FC Run`이라는 Docker-like CLI 도구로 오픈 소스화할 예정이다. 이는 Firecracker VM에서 컨테이너를 실행하고 스냅샷/복원하는 기능을 제공하며, 분당 15,000개의 VM 시작이 가능할 정도로 매우 빠르다. [[14:42](https://www.youtube.com/watch?v=svCnShDvgQg&t=882)], [[15:27](https://www.youtube.com/watch?v=svCnShDvgQg&t=927)]

*   **결론**
    *   컨텍스트 로그와 실행 스냅샷을 결합하면 버전, 턴, 실패에 걸쳐 내구성을 갖춘 에이전트를 만들 수 있으며, 이는 백엔드 인프라를 `stateless compute`에서 `stateful compute`로 전환하는 핵심이 될 것이다. [[16:00](https://www.youtube.com/watch?v=svCnShDvgQg&t=960)], [[11:10](https://www.youtube.com/watch?v=svCnShDvgQg&t=670)]

## [Hierarchical Memory: Context Management in Agents — Sally-Ann Delucia](https://www.youtube.com/watch?v=esY99nYXxR4)

**한 줄 인사이트**
💡 에이전트의 실패는 프롬프트 때문이 아니라 컨텍스트 관리 문제에서 비롯되며, 컨텍스트 엔지니어링, 메모리, 평가가 성공의 핵심이다.

**핵심 요약**
이 강연에서 Sally-Ann Delucia는 Arise의 AI 에이전트 Alex를 개발하면서 겪었던 컨텍스트 관리 문제와 해결책에 대해 공유합니다. 컨텍스트 윈도우의 한계를 넘어서기 위한 다양한 시도와 그 과정에서 얻은 교훈을 다룹니다.

*   **컨텍스트 관리의 중요성**
    *   작년 중반부터 '프롬프트 엔지니어링'보다 '컨텍스트 엔지니어링'의 중요성이 부각되기 시작했습니다. 에이전트의 성공 또는 실패는 컨텍스트에 의해 결정됩니다 [[2:02](https://www.youtube.com/watch?v=esY99nYXxR4&t=122)].
    *   컨텍스트 관리는 단순히 토큰 제한 내에 정보를 채워 넣는 것이 아니라, 모델이 보아야 할 가장 중요한 데이터를 전략적으로 선택하는 문제입니다 [[2:34](https://www.youtube.com/watch?v=esY99nYXxR4&t=154)].
    *   Alex는 Arise의 관측 플랫폼 위에 구축되어 AI 에이전트가 생성하는 방대한 트레이스 및 스팬 데이터를 처리해야 하므로, 컨텍스트 전략은 필수적이었습니다.

*   **실패한 컨텍스트 관리 전략**
    *   **단순 잘라내기(Naive Truncation)**: 처음 100자만 남기고 나머지를 버리는 방식은 에이전트가 이전 대화를 잊어버려 추론 능력을 저해했습니다. 예를 들어, 특정 입력에 대한 후속 질문을 이해하지 못했습니다 [[5:32](https://www.youtube.com/watch=v=esY99nYXxR4&t=332)].
    *   **요약(Summarization)**: LLM이 컨텍스트를 요약하도록 하는 방식은 어떤 정보가 중요한지 모델에 전적으로 맡기게 되어 일관성이 없고 신뢰할 수 없었습니다 [[6:15](https://www.youtube.com/watch=v=esY99nYXxR4&t=375)].

*   **성공적인 컨텍스트 관리 전략: 스마트 잘라내기 메모리**
    *   현재 Alex에서 사용되는 전략은 대화의 시작(head) 100자와 끝(tail) 100자를 유지하고, 중간 부분은 잘라내어 메모리에 저장하는 방식입니다 [[6:46](https://www.youtube.com/watch=v=esY99nYXxR4&t=406)].
    *   에이전트는 필요할 때 메모리 저장소에서 이전 컨텍스트를 검색할 수 있어, 중요한 정보에 대한 접근성을 유지하면서 컨텍스트 윈도우를 효율적으로 관리합니다.
    *   이 전략은 몇 달간 성공적으로 작동했으며, 컨텍스트는 모델이 보는 것을 결정하고 메모리는 살아남는 것을 결정합니다 [[7:47](https://www.youtube.com/watch=v=esY99nYXxR4&t=467)].

*   **긴 세션 관리 및 서브 에이전트 활용**
    *   사용자들이 채팅을 다시 시작하지 않고 긴 대화를 이어가는 경향이 있어, 컨텍스트가 계속 증가하고 에이전트의 실패가 뒤늦게 발견되는 문제가 있었습니다 [[8:03](https://www.youtube.com/watch=v=esY99nYXxR4&t=483)].
    *   **긴 세션 평가(Long Session Evals)**: 10번의 대화 턴을 로드한 후 11번째 턴을 테스트하여 컨텍스트 관리의 성능을 평가함으로써 버그를 조기에 발견합니다 [[9:00](https://www.youtube.com/watch=v=esY99nYXxR4&t=540)].
    *   **서브 에이전트(Sub-Agents)**: 모든 컨텍스트가 하나의 에이전트에 속할 필요는 없다는 깨달음에서, 검색과 같은 데이터 집약적인 무거운 작업은 서브 에이전트에 위임하여 메인 에이전트의 컨텍스트를 가볍게 유지합니다 [[10:00](https://www.youtube.com/watch=v=esY99nYXxR4&t=600)].

## [You can't just one shot it — Mehedi Hassan, Granola](https://www.youtube.com/watch?v=ON5LIT0M4do)

**한 줄 인사이트**
💡 LLM 기반 제품 개발은 한 번에 성공하기 어렵고, 사용자 경험을 개선하기 위해 LLM 동작을 추적하고 피드백 루프를 구축하는 것이 핵심입니다.

**핵심 요약**
Granola의 제품 엔지니어 Mehedi Hassan은 LLM 기반 기능을 프로덕션에 배포할 때 겪는 어려움과 이를 극복하기 위한 Granola의 접근 방식을 공유합니다. LLM 기능은 개발 환경에서는 쉽게 구현되는 것처럼 보이지만, 실제 서비스에서는 예상치 못한 문제에 직면하게 됩니다.

1.  **LLM 기능 배포의 어려움**
    *   **일반적인 챗봇의 한계**: Granola는 회의록 앱에 챗봇 기능을 추가했을 때, "도시 목록 제공" 요청에 웹 검색이 느리거나, 팔로우업 이메일 작성이 부자연스럽고, "회의 코칭" 요청에 엉뚱한 "축구 코치"를 언급하는 등 일반적인 챗봇의 문제점을 겪었습니다 [[02:20](https://www.youtube.com/watch?v=ON5LIT0M4do&t=140)].
    *   **웹 검색 도구의 복잡성**: LLM 제공업체에서 웹 검색 도구를 한 줄 코드로 쉽게 추가할 수 있다고 홍보하지만, 실제로는 토큰 사용량과 비용이 급증하고 (예: 채팅당 10펜스), 제공업체의 업데이트로 인해 성능이 예고 없이 저하되는 등 통제 불가능한 문제가 발생할 수 있습니다 [[03:12](https://www.youtube.com/watch?v=ON5LIT0M4do&t=192)].
    *   **출력 개인화의 난점**: 하나의 프롬프트로는 모든 사용자의 요구를 충족할 수 없습니다. 영업팀은 거래 요약, 엔지니어링팀은 액션 아이템/블로커/티켓, HR팀은 완전히 다른 것을 기대하는 등 역할에 따라 원하는 출력이 다르기 때문입니다 [[04:09](https://www.youtube.com/watch?v=ON5LIT0M4do&t=249)].

2.  **Granola의 해결책: 피드백 루프 구축**
    *   **내부 트레이싱 도구 개발**: Granola는 LLM의 '블랙박스' 문제를 해결하기 위해 자체 트레이싱 도구를 구축했습니다. 이 도구는 툴 호출, 검색 경로, 추론 과정, 비용 구조 등 LLM 파이프라인의 전체 과정을 시각화하여 엔지니어뿐만 아니라 제품, 데이터, CX팀 등 모든 내부 직원이 CloudWatch 같은 복잡한 쿼리 없이 문제를 파악하고 개선할 수 있도록 돕습니다 [[04:39](https://www.youtube.com/watch?v=ON5LIT0M4do&t=279), [05:13](https://www.youtube.com/watch?v=ON5LIT0M4do&t=313)].
    *   **데스크톱 앱의 웹 앱 전환**: Electron 기반 데스크톱 앱의 특성상 여러 기능 변형을 동시에 테스트하기 어려웠던 문제를 해결하기 위해, Electron 앱의 프런트엔드를 웹 셸로 전환하여 온라인에 배포했습니다 [[06:45](https://www.youtube.com/watch=ON5LIT0M4do&t=405), [07:16](https://www.youtube.com/watch=ON5LIT0M4do&t=436)].

결론적으로, LLM 기반 제품 개발의 핵심은 '한 번에 완벽하게 만드는 것'이 아니라, LLM과 테니스 게임을 하듯이 지속적인 피드백 루프를 통해 제품을 개선하고 사용자에게 마법 같은 경험을 제공하는 것입니다 [[08:48](https://www.youtube.com/watch=ON5LIT0M4do&t=528)].
