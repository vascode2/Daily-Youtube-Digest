## [Comprehend First, Code Later: The AI Skill I Rely On Daily — Priscila Andre de Oliveira, Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM)

**한 줄 인사이트**
💡 대규모 코드베이스에서 AI 협업의 핵심은 코드 생성이 아닌 '코드 이해(Comprehension)'이며, 개발자와 AI 간의 올바른 멘탈 모델 정립이 오작동을 막는 열쇠입니다.

**핵심 요약**
Sentry의 소프트웨어 엔지니어 Priscila Andre de Oliveira는 자신이 사용한 116개의 Claude 세션을 분석하여 AI 활용의 67%가 코드 생성이 아닌 코드 이해에 쓰였다는 사실을 공유합니다. 그녀는 하루에 약 100개의 PR이 병합되는 복잡한 환경에서 개발자가 AI에게 무작정 코드를 짜게 하기보다 시스템을 먼저 파악하는 것이 중요하다고 주장합니다. 이를 해결하기 위해 아키텍처, 컨벤션 등 6가지 탐색 모드를 갖춘 개인용 AI 도구 'Catch Me Up'을 구축하여 올바른 이해를 바탕으로 한 협업 프로세스를 제시합니다.

1. **대규모 코드베이스에서의 AI 활용 분석**
   - Sentry에서 근무하는 Priscila는 자신의 Claude 사용 이력 116개 세션을 분석한 결과, 코드 생성은 단 2%에 불과했고 67%가 코드 이해를 위한 것이었음을 발견했습니다.
   - 15년의 역사와 매일 100개의 PR이 병합되는 대규모 환경에서는 단순히 코드를 빠르게 짜는 것보다 기존 시스템을 정확히 파악하는 것이 훨씬 더 중요합니다.

2. **이해 중심의 AI 워크플로우와 'Catch Me Up' 도구**
   - 개발자가 시스템을 오해한 상태에서 AI에게 코딩을 시키면 잘못된 결과물(slop)이 나오기 때문에, 계획과 구현 전에 철저한 이해 단계가 선행되어야 합니다.
   - 이를 위해 아키텍처, 컨벤션, 기능 추적, 구문, 테스트, 히스토리 등 6가지 탐색 모드를 지원하는 개인용 AI 스킬 'Catch Me Up'을 개발하여 활용하고 있습니다.

3. **AI 에이전트 협업에서의 멘탈 모델 정립**
   - AI 에이전트에게 작업을 위임하기 전에 에이전트가 찾아낸 맥락을 개발자가 먼저 검토하고 동기화하는 루프를 거쳐야 정교한 계획 수립이 가능합니다.
   - 개발자와 AI 간의 정렬되지 않은 멘탈 모델은 결국 기술 부채와 코드 품질 저하로 이어지므로, 이해를 돕는 AI 도구의 역할이 더욱 강조됩니다.

## [Why Rust is the Ideal Language for Vibe-Coding — Daniel Szoke, Sentry](https://www.youtube.com/watch?v=ugUeZ8-b-u0)

**한 줄 인사이트**
💡 AI 에이전트 코딩 환경에서는 쓰기 쉬운 언어보다 컴파일러가 엄격한 제약 조건과 즉각적인 피드백을 제공하는 Rust가 더 안전하고 효율적입니다.

**핵심 요약**
Sentry의 Rust SDK 메인테이너인 Daniel Szoke는 AI 모델이 작성하기 쉬운 Python이나 TypeScript가 실제로는 컴파일러의 제약이 없어 치명적인 런타임 버그를 유발한다고 지적합니다. 그는 AI 에이전트가 코드를 작성할 때 컴파일러가 오류를 엄격하게 강제하는 환경이 오히려 더 빠르고 안전한 개발 루프를 만든다고 주장합니다. 이를 통해 에이전트가 컴파일러의 피드백을 받아 스스로 오류를 수정함으로써 프로덕션 버그의 배포를 원천 차단할 수 있습니다.

1. **쉬운 언어가 초래하는 런타임 버그의 위험성**
   - TypeScript나 Python은 제약이 적어 AI가 코드를 빠르게 생성하도록 돕지만, 동시에 컴파일러가 감지하지 못하는 간헐적인 데이터 경쟁 버그를 남깁니다.
   - 이러한 버그는 런타임 중에만 무작위로 발생하기 때문에 인간 개발자가 사후에 디버깅하고 수정하기가 매우 까다롭습니다.
2. **Rust 컴파일러의 강력한 피드백 루프**
   - Rust는 스레드 안전성 문제가 있을 때 컴파일을 거부하며, 컴파일러가 안전하지 않은 타입을 지목하고 구체적인 해결책을 에이전트에게 직접 제시합니다.
   - AI 에이전트는 이 명확한 컴파일러 피드백을 루프 삼아 스스로 코드를 수정할 수 있어, 인간이 코드를 리뷰하는 것보다 빠르고 안정적입니다.
3. **AI 에이전트의 성능을 극대화하는 제약 조건**
   - 자율형 AI 에이전트에게는 무제한의 자유보다 엄격한 규칙과 제약 조건이 주어질 때 더 일관되고 안전한 결과물을 도출할 수 있습니다.
   - 컴파일러가 에이전트의 행동 반경을 제한하는 안전장치 역할을 수행하므로 잘못된 코드가 프로덕션에 배포되는 일을 방지합니다.

## [The maturity phases of running evals — Phil Hetzel, Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)

**한 줄 인사이트**
💡 LLM 평가는 모든 실패를 방지하는 단위 테스트가 아니라, 알려진 실패 모드를 정의하고 프로덕션 데이터를 피드백 루프로 연결하여 점진적으로 개선하는 플라이휠 모델로 접근해야 합니다.

**핵심 요약**
Braintrust의 Phil Hetzel은 많은 개발 팀이 LLM 평가(evals)를 모든 실패를 완벽히 막으려는 단위 테스트처럼 잘못 접근하고 있다고 지적합니다. 그는 알려진 실패 모드를 구체적으로 정의해 빠르게 배포한 뒤, 실제 프로덕션 트레이스를 오프라인 실험으로 피드백하는 플라이휠 구축이 핵심이라고 주장합니다. 이를 위해 단순한 정성적 확인부터 외부 시스템과 연동되는 CRUD 도구 평가까지 LLM 평가의 4단계 성숙도 모델과 구체적인 해결 방안을 제시합니다.

1. **단위 테스트 관점의 탈피와 피드백 플라이휠 구축**
   - LLM 평가는 모든 예외 상황을 미리 막으려는 단위 테스트가 아니며, 알려진 실패 모드를 명확히 규정하고 빠르게 제품을 배포하는 것이 우선입니다.
   - 실제 프로덕션 환경의 트레이스(Trace) 데이터를 수집하여 오프라인 실험으로 환류시키고, 이를 통해 다음 개선을 이끄는 선순환 플라이휠을 만들어야 합니다.

2. **인간의 정성적 평가에서 LLM 판사(LLM as Judge)로의 진화**
   - 평가의 시작 단계인 '바이브 체크(Vibe checking)'에서는 단순한 추천/비추천을 넘어 인간 평가자의 구체적인 정성적 근거를 문서화해야 합니다.
   - 이렇게 축적된 인간의 평가 근거와 정당성 데이터를 기반으로 삼아, 대규모 환경에서 일관되게 작동하는 LLM 판사(LLM as Judge) 시스템을 구축할 수 있습니다.

3. **외부 시스템 연동 및 CRUD 도구 평가의 난제 해결**
   - 단순히 정보를 조회하는 컨텍스트 수집 도구와 달리, 외부 시스템의 상태를 변경하는 CRUD 도구는 평가 시점에 원래 트레이스가 실행된 정확한 순간의 상태를 재현해야 하므로 평가 난이도가 매우 높습니다.
   - 이를 해결하기 위해 벡터 데이터베이스에 대한 타임스탬프 쿼리를 수행하거나, 캡처된 시스템 상태를 트레이스에 직접 주입하는 두 가지 기술적 접근법을 활용할 수 있습니다.

## [Finally a good benchmark (DeepSWE)](https://www.youtube.com/watch?v=6LwQ8RbU9as)

**한 줄 인사이트**
💡 DeepSWE는 기존 SWE-bench의 데이터 오염 문제를 해결하고, AI 모델의 실제 소프트웨어 엔지니어링 능력을 정밀하게 측정할 수 있는 새로운 벤치마크 표준을 제시합니다.

**핵심 요약**
IT 크리에이터 매튜 버먼은 기존 AI 코드 생성 평가 도구들의 한계를 지적하며, DataCurve가 개발한 새로운 벤치마크인 DeepSWE를 소개합니다. 그는 기존 SWE-bench가 학습 데이터 오염과 단순 코드 수정에 치우쳐 있다는 점을 비판하고, DeepSWE가 어떻게 실제 개발 환경과 유사한 복잡한 문제를 출제하는지 분석합니다. 이를 통해 AI 에이전트의 진정한 문제 해결 능력을 평가하기 위한 새로운 기준과 그 필요성을 역설합니다.

1. **기존 SWE-bench의 데이터 오염 및 한계**
   - 기존의 SWE-bench는 테스트 세트가 인터넷에 공개되어 있어 LLM의 학습 데이터에 포함되는 '데이터 오염' 문제에 취약하다는 치명적인 단점이 있습니다.
   - 단순히 깃허브 이슈를 해결하는 방식은 실제 소프트웨어 엔지니어가 겪는 복잡한 아키텍처 설계나 다중 파일 분석 능력을 온전히 평가하기 어렵습니다.

2. **DeepSWE가 제안하는 새로운 평가 방식**
   - DeepSWE는 데이터 오염을 방지하기 위해 지속적으로 업데이트되는 비공개 테스트 케이스와 실시간으로 생성되는 개발 과제를 활용합니다.
   - 단순한 코드 수정을 넘어 전체 코드베이스의 의존성을 이해하고, 복잡한 논리적 오류를 스스로 디버깅하는 능력을 중점적으로 측정합니다.

3. **실제 개발 환경을 모사한 샌드박스 평가**
   - 평가 대상 AI 모델은 격리된 보안 샌드박스 환경에서 실제로 코드를 실행하고 테스트를 통과해야만 점수를 획득할 수 있습니다.
   - 이를 통해 모델이 단순히 그럴듯한 코드를 생성하는 것을 넘어, 실제로 작동하고 배포 가능한 수준의 소프트웨어를 빌드할 수 있는지 검증합니다.

4. **AI 에이전트 성능 평가의 새로운 표준**
   - DeepSWE는 Claude 3.5 Sonnet이나 GPT-4o와 같은 최신 프론티어 모델들의 실제 코딩 역량 격차를 명확하게 구분해 줍니다.
   - 개발자들과 기업들은 이 벤치마크를 통해 자사의 워크플로우에 가장 적합한 AI 코딩 에이전트를 객관적인 지표로 선택할 수 있게 됩니다.
