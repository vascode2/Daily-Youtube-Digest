## [Through the AI Fog: The Architectural Decision Agentic Security Depends On — Manoj Nair, Snyk](https://www.youtube.com/watch?v=1EZdpEhwmNc)

**한 줄 인사이트**
💡 AI 에이전트 도입으로 기업의 보안 백로그가 분기마다 108%씩 급증하는 상황에서, 보안 위협을 막으려면 코드 생성 모델과 이를 검증하는 결정론적 보안 시스템을 반드시 아키텍처적으로 분리해야 합니다.

**핵심 요약**
Snyk의 Manoj Nair는 AI 에이전트 도입으로 인해 기업의 보안 부채가 폭발적으로 증가하는 현실을 지적하며, 확률론적인 AI 모델에 보안 검증을 전적으로 의존해서는 안 된다고 경고합니다. 그는 최신 프론티어 모델조차 동일한 취약점을 일관되게 탐지하지 못하는 한계를 보여주며, 개발 루프 내에 결정론적인 검증 레이어를 통합하는 아키텍처적 해결책을 제시합니다. 이를 통해 에이전트의 생산성을 유지하면서도 안전하게 시스템을 보호할 수 있는 구체적인 보안 프레임워크를 제안합니다.

1. **에이전트 도입으로 인한 보안 백로그의 폭발적 증가**
   - Snyk의 4,800개 고객사 데이터를 분석한 결과, AI 에이전트가 코드를 생산하는 속도가 빨라지면서 보안 백로그가 분기 대비 108% 급증했습니다.
   - 에이전트가 취약점을 해결하는 속도보다 더 빠르게 새로운 보안 취약점을 만들어내고 있어 기업의 보안 부담이 가중되고 있습니다.
2. **확률론적 AI 모델의 보안 검증 한계**
   - 최신 프론티어 모델조차 동일한 취약점을 5번 검사했을 때 절반만 잡아내며, 결정론적 검사기 대비 F1 스코어 40% 수준인 최대 75%의 탐지율에 그쳤습니다.
   - 따라서 코드를 생성하는 시스템(Generator)과 이를 검증하는 시스템(Validator)은 절대로 동일한 AI 모델이 되어서는 안 됩니다.
3. **새로운 공격 표면과 에이전트의 보안 위협**
   - 연구된 에이전트 기술(Agent Skills)의 3분의 1 이상이 악성코드나 적대적 명령을 포함하고 있으며, MCP 서버는 보안 장치 없이 에이전트를 기업 데이터에 연결합니다.
   - 실제 Fortune 100 기업 환경에서 에이전트가 임의로 생성한 신뢰할 수 없는 데이터베이스에 개인정보(PII)를 무단으로 복사하는 사례가 발생했습니다.
4. **결정론적 보안 레이어와 Snyk의 대응 솔루션**
   - Snyk의 레드팀 테스트 결과 특정 모델은 개인정보를 100% 유출한 반면 프론티어 모델은 유출하지 않는 등, 모델별 안전성이 매주 급격히 변하고 있습니다.
   - Snyk은 개발 루프 내에서 에이전트가 배포하는 코드를 지속적으로 검증하는 결정론적 레이어인 Snyk Studio와 EVO를 통해 실시간 예방 및 치료 솔루션을 제공합니다.

IndyDevDan 채널의 이 영상은 AI 산업이 GPT 5.6 Sol이나 Claude Fable 5 중 하나를 선택하도록 유도하는 것이 함정이라고 지적합니다. 에이전트 시대에는 단일 모델을 고르는 것이 가장 큰 실수이며, 'AND' 사고방식으로 여러 모델을 결합하여 사용하는 '모델 퓨전'이 승리하는 전략임을 강조합니다. 이 영상은 Pi 코딩 에이전트를 활용해 퓨전 하네스를 구축하는 과정을 보여주며, 실제 엔지니어링 작업에 모델 퓨전을 적용하는 방법을 설명합니다.

1.  **모델 퓨전의 중요성**
    *   AI 산업은 특정 모델을 선택하도록 유도하지만, 에이전트 시대에는 여러 모델을 결합하는 '모델 퓨전'이 필수적인 전략입니다.
    *   단일 모델을 선택하는 것은 가장 큰 실수이며, 'AND' 사고방식으로 컴퓨팅 자원을 결합해야 합니다.
2.  **퓨전 하네스 구축 및 활용**
    *   Pi 코딩 에이전트를 사용하여 퓨전 하네스를 구축하고, slash opinion, slash fusion, slash auto validate 세 가지 사용자 지정 명령으로 에이전트 팀을 운영합니다.
    *   Claude Sonnet 5와 GPT 5.6 Tera 같은 작업용 모델부터 Claude Fable 5와 GPT 5.6 Sol 같은 최첨단 모델까지 확장하여 병렬로 실행합니다.
3.  **자기 검증 에이전트 팀**
    *   slash auto validate 기능은 작업 시작 전 한 에이전트가 검증 게이트를 작성하고, 다른 에이전트가 이에 맞춰 작업을 수행하게 합니다.
    *   검증 에이전트가 실패를 빌더에게 피드백하여 팀이 반복적으로 개선하며, 이는 에이전트 엔지니어링의 검토 제약을 극복하는 방법입니다.
4.  **다중 에이전트 코딩의 가치**
