## [Building Interactive UIs in VS Code with MCP Apps — Marlene Mhangami & Liam Hampton, GitHub](https://www.youtube.com/watch?v=_xIwFcnHqp4)

**한 줄 인사이트**
💡 VS Code 채팅창 내에서 단순 텍스트나 링크가 아닌, MCP(Model Context Protocol)를 활용해 안전하게 격리된 대화형 HTML UI를 직접 렌더링하고 실시간으로 제어할 수 있습니다.

**핵심 요약**
GitHub의 Marlene Mhangami와 Liam Hampton은 VS Code 채팅창 내에서 단순한 텍스트 답변을 넘어 동적인 대화형 UI를 구현하는 방법을 소개합니다. 이들은 모델 컨텍스트 프로토콜(MCP) 도구를 활용해 샌드박스화된 iframe 형태로 라이브 UI를 렌더링하고 서버와 실시간으로 데이터를 주고받는 메커니즘을 설명합니다. 실제 Shopify의 결제 흐름이나 Excalidraw의 다이어그램 편집 사례를 바탕으로, 개발자가 MCP 리포지토리의 기능을 사용해 이를 처음부터 빌드하는 과정을 단계별로 보여줍니다.

1. **MCP를 통한 대화형 UI 렌더링 메커니즘**
   - MCP 도구는 데이터와 함께 번들링된 HTML UI를 가리키는 리소스 참조를 반환하며, VS Code는 이를 채팅창 내의 샌드박스 iframe으로 가져와 렌더링합니다.
   - 사용자가 UI와 상호작용하면 앱이 서버를 다시 호출하고, 서버가 새로운 데이터를 반환하여 화면을 실시간으로 업데이트하는 구조입니다.

2. **안전한 실행을 위한 샌드박스 환경**
   - VS Code 설정이 손상되거나 외부 API가 무단으로 호출되는 것을 방지하기 위해, 렌더링되는 라이브 iframe은 철저하게 격리된 샌드박스 내에서 실행됩니다.
   - 이는 시스템을 보호하기 위해 안전장치를 두는 것과 같으며, 사용자가 안심하고 채팅창 내에서 다양한 인터랙션을 수행할 수 있도록 돕습니다.

3. **실제 기업들의 MCP UI 활용 사례**
   - Shopify는 사용자가 채팅창 내부에서 이탈하지 않고 바로 결제 단계를 진행할 수 있도록 이 대화형 UI 패턴을 도입해 활용하고 있습니다.
   - Excalidraw는 채팅창 안에서 사용자가 직접 요소를 드래그하고 편집할 수 있는 대화형 아키텍처 다이어그램 기능을 제공합니다.

4. **MCP 리포지토리를 활용한 실무 구현**
   - 개발자 Marlene Mhangami와 Liam Hampton은 Go 애플리케이션에서 버블 정렬과 피보나치 수열을 실행하고, 그 결과를 대화형 플레임 그래프로 시각화하는 데모를 선보입니다.
   - 이들은 MCP 리포지토리에 등록된 기술(skill)을 활용하여 개발자가 처음부터 이러한 대화형 앱을 빌드하는 구체적인 워크플로우를 안내합니다.

## [Evals Are Broken, Use Them Anyway — Ara Khan, Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)

**한 줄 인사이트**
💡 AI 에이전트의 성능 향상은 단순히 더 좋은 모델로 바꾸는 것이 아니라, 컨테이너 설정 최적화와 실패 경로에 대한 체계적인 분석을 통해 이루어집니다.

**핵심 요약**
Cline의 개발자 Ara Khan은 터미널 벤치마크(Terminal Bench)에서 성능을 43%까지 끌어올린 실전 경험을 바탕으로 기존 AI 평가 방식의 한계와 극복 방안을 제시합니다. 단순히 더 뛰어난 모델로 교체하는 것보다 시스템 내부의 세부 설정을 최적화하는 것이 중요하다고 주장하며, 실패 사례를 체계적으로 분석하는 실용적인 프레임워크를 제안합니다. 이를 통해 벤치마크 점수의 맹신과 막연한 직관 사이에서 균형을 잡고 실제 작동하는 시스템을 구축하는 구체적인 경로를 보여줍니다.

1. **모델 교체 없는 시스템 및 프롬프트 최적화**
   - Cline은 Terminal Bench에서 43%의 시작 점수를 기록했으며, 이는 모델 변경이 아닌 컨테이너 CPU 및 메모리 설정, 타임아웃 연장, Anthropic 모델 전용 프롬프트 엔지니어링을 통해 달성되었습니다.
   - 특정 모델 제품군에 맞춤화된 프롬프트 기술은 Codex나 Gemini 같은 다른 모델로 쉽게 전이되지 않으므로 고유한 최적화가 필요합니다.

2. **실패 분석을 위한 에이전트 활용 프레임워크**
   - 평가 실행 후 실패한 모든 흔적(trace)에 다른 에이전트를 투입하여 어떤 미세한 조정이 실제로 점수를 움직이는지 포트폴리오 방식으로 분류하고 분석해야 합니다.
   - 이 프로세스는 단순한 수치 비교를 넘어 시스템의 병목 구간을 명확히 파악하고 개선할 수 있는 실질적인 데이터를 제공합니다.

3. **실패 영역의 세 가지 분류(Zone) 체계**
   - 실패 원인은 명백한 버그(Zone 1), 훌륭하다고 평가받는 모델이 특정 테스트 환경에서 작동하지 않는 미묘한 차이(Zone 2), 그리고 지양해야 할 벤치마크 과적합(Zone 3)으로 나뉩니다.
   - 특히 벤치마크에만 과적합되는 현상을 경계하면서, 자신의 고유한 테스트 환경에 맞게 모델의 성능을 미세 조정하는 데 집중해야 합니다.
