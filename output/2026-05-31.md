**핵심 요약**

   - 기존 기판의 한계를 극복하기 위해 패키징 분야에서도 CoWoP(Chip-on-Wafer-on-Substrate) 같은 차세대 기술이 논의되며 기판과 패키징의 경계가 허물어지고 있습니다.

4. **국내 주요 PCB 및 소재 기업들의 기술 경쟁 구도**

## [Can LLMs generate Enterprise Quality Code? — Prasenjit Sarkar, Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)

**한 줄 인사이트**
💡 LLM이 생성하는 코드의 양이 급증하면서 보안 취약점 비율도 함께 높아지고 있으며, 이를 해결하기 위해 커밋 전 단계부터 에이전트 기반의 자동 검증 및 수정 프레임워크(ACDC)를 도입해야 합니다.

**핵심 요약**
Sonar의 Prasenjit Sarkar는 53개 LLM 모델을 대상으로 4,444개의 Java 프로그래밍 과제를 테스트하여 생성된 코드의 품질과 보안 취약점을 정밀하게 분석했습니다. 그는 AI가 생성한 코드의 버그와 보안 문제를 해결하기 위해 개발 주기 전반에서 작동하는 3단계 'ACDC(Guide, Verify, Solve)' 프레임워크를 제시합니다. 이를 통해 개발자가 코드를 커밋하기 전에 신속하게 분석을 수행하고, PR 단계에서 자동화된 에이전트가 검증된 수정안을 제공하는 워크플로우를 제안합니다.

1. **LLM 코드 생성 벤치마크 결과**
   - Sonar가 53개 모델을 통해 4,444개의 Java 과제를 수행한 결과, GPT-4o는 25만 라인 미만을 생성한 반면 GPT 5.4는 120만 라인을 생성했습니다.
   - Claude Sonnet 4.6은 62만 7,000라인을 생성했으나, 100만 라인당 300개의 보안 문제가 발생하여 가장 높은 보안 취약점 비율을 기록했습니다.
2. **3단계 ACDC 프레임워크 제안**
   - AI 코드의 품질 관리를 위해 Sonar는 가이드(Guide), 검증(Verify), 해결(Solve)로 구성된 3단계 ACDC 프레임워크를 구축했습니다.
   - 이 프레임워크는 개발자가 코드를 작성하는 시점부터 최종 풀 리퀘스트(PR) 단계까지 실시간으로 코드의 복잡도와 버그를 제어합니다.
3. **실시간 검증 및 자동 수정 에이전트**
   - 검증(Verify) 단계에서는 기존 CI 환경에서 1~5분 걸리던 SonarQube 분석을 커밋 전에 단 1~5초 만에 실행하여 피드백 루프를 극대화합니다.
   - PR 단계로 넘어간 오류는 자동 수정 에이전트가 이슈당 하나의 수정안을 생성하고, 컴파일 및 회귀 테스트를 거쳐 검증된 해결책만 개발자에게 제시합니다.

## [Engineering voice agents: Latency, quality, and scale — Rishabh Bhargava, Together AI](https://www.youtube.com/watch?v=N7b1PJc7SFc)

**한 줄 인사이트**
💡 음성 에이전트의 사용자 이탈을 막기 위해서는 네트워크 공동 배치를 통해 지연 시간을 5ms 수준으로 낮추고, 8B~30B 규모의 최적화된 LLM 파이프라인을 구축해야 합니다.

**핵심 요약**
Together AI의 Rishabh Bhargava는 사용자가 대화를 중단하지 않는 자연스러운 음성 에이전트를 구현하기 위한 기술적 아키텍처를 제시합니다. 그는 음성 인식(STT), LLM, 음성 합성(TTS) 파이프라인 전반에서 발생하는 지연 시간을 밀리초(ms) 단위로 분석하고, 이를 극복하기 위한 실질적인 네트워크 및 모델 최적화 전략을 설명합니다. 특히 대기 시간을 줄이면서도 복잡한 도구 호출을 수행할 수 있는 이중 LLM 구조의 유용성을 강조합니다.

1. **지연 시간 단축을 위한 네트워크 공동 배치**
   - 서로 다른 데이터 센터에 위치한 모델 간의 75ms 네트워크 지연은 전체 파이프라인에 30%의 오버헤드를 추가하므로, 모든 요소를 한 건물에 공동 배치하여 이를 5ms 수준으로 단축해야 합니다.

2. **음성 에이전트에 최적화된 LLM 크기 선정**
   - 첫 번째 토큰 생성 시간(TTFT)을 200~300ms로 유지하기 위해 8B에서 30B 매개변수 범위의 LLM을 사용해야 하며, 너무 큰 모델은 지연 시간을 초과하고 너무 작은 모델은 도구 호출 성능이 떨어집니다.

3. **STT 성능 기준과 효율적인 워크플로우 패턴**
   - 음성 인식(STT) 단계의 P90 지연 시간을 100ms 미만으로 유지하고 단어 오류율(WER)을 약 6% 수준으로 통제하는 것이 고품질 서비스의 기준이 됩니다.
   - 대화 흐름을 관리하는 소형 LLM이 빠른 응답을 담당하고, 복잡한 요청이 발생할 때만 대형 모델에 단일 도구 호출을 보내는 패턴을 통해 속도와 정확성을 모두 잡을 수 있습니다.

## [Spec-Driven Testing for Agents With A Brain the Size of A Planet — Steven Willmott, SafeIntelligence](https://www.youtube.com/watch?v=UQKg0td-Bf4)

**한 줄 인사이트**
💡 모델의 크기가 커질수록 보안 취약점과 공격 표면도 함께 늘어나므로, 모델 교체 시에도 유지되는 명확한 명세(Spec) 기반의 검증 체계를 구축해야 합니다.

**핵심 요약**
Safe Intelligence의 Steven Willmott는 거대 모델이 가진 넓은 기능성이 오히려 더 큰 보안 위협과 공격 표면을 만들어낸다고 경고합니다. 그는 단순한 테스트 데이터셋 구축을 넘어 에이전트의 규칙, 도메인 온톨로지, 역할 권한 등을 독립적으로 정의하는 '명세 기반 검증(Spec-Driven Validation)'의 필요성을 역설합니다. 이를 통해 개발자는 모델을 변경하더라도 일관된 보안 및 성능 테스트를 수행하고 지속적인 개선을 이끌어낼 수 있습니다.

1. **거대 모델의 역설과 증가하는 보안 위협**
   - 모델의 크기가 커지고 능력이 다양해질수록 악의적인 지시를 시(poem)로 포장하는 등의 고차원적인 탈옥(Jailbreak) 공격에 더 취약해집니다.
   - 소형 모델은 복잡한 맥락을 이해하지 못해 공격을 실행하지 않지만, 대형 모델은 이를 이해하고 실행하여 더 넓은 인프라 권한을 남용할 위험이 있습니다.

2. **명세 기반 검증(Spec-Driven Validation)의 정의**
   - 에이전트 명세는 단순한 테스트 데이터셋에 그치지 않고, '10% 이상의 할인 제공 금지'와 같은 명확한 규칙과 도메인 온톨로지를 포함해야 합니다.
   - 예를 들어 항공사 에이전트의 경우 해당 항공사가 실제로 취항하는 목적지만 다루도록 도메인 영역을 엄격하게 제한해야 합니다.

3. **역할 정의 및 견고성 요구사항 수립**
   - 에이전트가 수행할 수 있는 권한과 역할을 명확히 규정하고, 오탈자나 다양한 문장 재표현(Rephrasing)에도 오작동하지 않는 견고성 기준을 마련해야 합니다.
   - 이러한 명세는 구현체와 독립적으로 작성되어야 모델을 다른 종류로 교체하더라도 보안 테스트와 반복적인 개선 프로세스를 안정적으로 유지할 수 있습니다.

## [실리콘밸리 VC 교과서 (A16Z)](https://www.youtube.com/watch?v=yRVj8NO7D0g)

**한 줄 인사이트**
💡 실리콘밸리 최고의 VC인 a16z의 벤 호로위츠는 자본의 성격이 바뀐 AI 시대에 기업이 생존하려면 의사결정 권한의 중앙 집중화와 눈앞의 구체적인 문제 해결에 집중해야 한다고 강조합니다.

**핵심 요약**
세계적인 벤처캐피털 a16z의 공동 창업자 벤 호로위츠는 기존 VC 업계의 한계를 극복하고 혁신을 이뤄낸 자신들의 경영 철학을 공유합니다. 그는 의사결정 구조의 효율화부터 AI 시대의 자본 레이스 변화, 그리고 성공적인 창업가가 갖춰야 할 실질적인 자세를 인터뷰를 통해 구체적으로 밝힙니다.

1. **창업자를 위한 '더 나은 제품'으로서의 VC**
   - 2009년 창업 당시 기존 VC들은 투자자에게만 유리하고 창업자에게는 자금 외에 도움을 주지 못하는 구조였습니다. a16z는 이러한 문제를 해결하고 창업자에게 실질적인 성장을 지원하는 서비스를 제공하고자 설립되었습니다.

2. **의사결정 권한의 중앙 집중화를 통한 빠른 확장**
   - 기존 VC의 평등한 파트너십 구조는 변화를 막는 장애물이 되었기에, a16z는 경제적 이익은 공유하되 의사결정 권한을 중앙 집중화했습니다. 이를 통해 크립토, 바이오 등 새로운 영역으로 신속하게 조직을 확장하고 스케일업할 수 있었습니다.

3. **AI 시대에 변화된 자본 레이스의 성격**
   - 과거 소프트웨어 시대에는 소통 비용 때문에 자본을 밀어넣는 방식이 불가능했지만, AI 시대에는 충분한 GPU와 데이터만 있으면 자본 투입으로 순식간에 기술 격차를 좁힐 수 있게 되었습니다.

4. **거창한 계획보다 눈앞의 구체적인 문제 해결**
   - 마크 저커버그나 드롭박스의 드류 휴스턴처럼 위대한 기업들은 처음부터 거창한 피치덱을 들고 시작하기보다 당장 눈앞의 하드코어한 문제를 풀다가 탄생했습니다.

5. **강력한 리더십을 바탕으로 한 신속한 의사결정**
   - 기업은 민주주의 조직이 아니며, 치열한 비즈니스 전장에서 살아남기 위해서는 모든 결정을 투표에 부치기보다 단판을 지어줄 강력한 리더가 존재해야 승리할 수 있습니다.

## [생각 그만하고 좀 만들어라 (트위터 창업자 잭도시)](https://www.youtube.com/watch?v=eLp3ocXamyY)

**한 줄 인사이트**
💡 남들의 기준을 따르기보다 반직관적인 아이디어를 철저한 실천으로 증명해내는 것이 진정한 혁신을 만드는 길입니다.

**핵심 요약**
트위터와 스퀘어의 창업자 잭 도시는 모두가 복잡한 기능을 더하고 가상 세계에 집중할 때, 오히려 단순함과 오프라인 결제라는 반직관적인 선택으로 세상을 바꾸었습니다. 그는 남들의 목소리에 흔들리지 않고 스스로 기준을 세우며 끊임없이 무언가를 만들어내는 실천의 중요성을 강조합니다. 과거의 혁신에 안주하지 않고 비트코인과 AI 네이티브 조직을 실험하는 그의 행보를 통해 실행력의 진정한 가치를 되짚어봅니다.

1. **반직관적인 선택과 단순함의 미학**

   - 모두가 가상 세계와 온라인에만 주목할 때, 오프라인 결제 단말기인 스퀘어(Square)를 개발하며 현실 세계의 문제를 직접 해결했습니다.

2. **스스로 기준을 만드는 실천적 경영**

   - 생각에만 머무르지 않고 철저하게 실행과 결과물로 자신의 가설을 증명해 나가는 실천 중심의 철학을 보여줍니다.

3. **끊임없는 미래 기술의 실험**
