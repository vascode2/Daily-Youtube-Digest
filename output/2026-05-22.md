## [Google Was Everywhere... Rewriting the Internet Order with Gemini](https://www.youtube.com/watch?v=jH4pZdrn-oU)

**한 줄 인사이트**
💡 구글은 단순한 챗봇 경쟁을 넘어 검색, 메일, 쇼핑 등 기존 서비스를 하나의 AI 실행 레이어로 통합하여 사용자를 대신해 작업을 수행하는 새로운 인터넷 질서를 구축하고자 합니다.

**핵심 요약**
안될공학은 Google I/O 2026에서 공개된 구글의 AI 전략이 단순한 모델 성능 개선을 넘어 인터넷 생태계 자체를 재편하려는 시도라고 분석합니다. 구글은 검색, Gmail, 안드로이드 등 기존의 방대한 서비스들을 하나의 AI 실행 레이어로 통합하여 사용자가 앱 사이를 이동할 필요가 없는 환경을 만들고자 합니다. 이를 위해 제미나이(Gemini) 라인업을 고도화하고 쇼핑과 결제를 연결하는 새로운 인프라를 제시하며 AI가 직접 행동하는 에이전트 시대로의 전환을 선언했습니다.

1. **모든 서비스를 하나로 묶는 AI 실행 레이어 통합**
   - 구글은 검색, Gmail, Docs, YouTube, 쇼핑, 안드로이드, 클라우드를 개별 앱이 아닌 하나의 거대한 AI 실행 레이어로 통합하는 전략을 추진합니다.

2. **작업 수행에 최적화된 제미나이 모델 라인업 고도화**
   - Gemini 3.5 Flash는 단순한 빠른 응답을 넘어 반복적인 실행과 장기적인 작업을 수행하는 에이전트 엔진 역할을 맡습니다.
   - Gemini Omni는 단순 영상 제작보다 더 까다로운 병목 구간인 반복적인 편집(Repetitive Editing) 작업을 표적으로 삼아 효율성을 극대화합니다.

3. **사용자 워크플로우를 대행하는 Gemini Spark**
   - Gemini Spark는 사용자가 이메일, 문서, 캘린더, 쇼핑 앱 등을 직접 오가며 수동으로 처리하던 복잡한 워크플로우를 AI에게 완전히 위임하도록 돕습니다.

4. **에이전트 쇼핑과 결제를 위한 신규 인프라 구축**

   - 이는 단순한 챗봇 대화를 넘어 AI가 사용자의 허락 하에 앱과 서비스를 넘나들며 실질적인 구매와 작업을 대행하는 미래 인터넷의 기반이 됩니다.

3. **다중 에이전트를 활용한 대규모 병렬 컴포넌트 생성**

## [Lobster Trap: OpenClaw in Containers — Sally Ann O'Malley, Red Hat](https://www.youtube.com/watch?v=F1DYkY1BlfM)

**한 줄 인사이트**
💡 AI 에이전트 환경을 컨테이너화하여 Podman과 Kubernetes로 배포하면 복잡한 설정 없이 2초 만에 동일한 실행 환경을 복제하고 팀 표준으로 확장할 수 있습니다.

**핵심 요약**
Red Hat의 Sally Ann O'Malley는 복잡한 YAML 파일과 설정 문서 대신 컨테이너 이미지를 활용해 AI 에이전트 환경을 표준화하는 방법을 제안합니다. 로컬 환경의 Podman에서 단 2초 만에 하위 에이전트를 실행하고 이를 Kubernetes로 확장함으로써, 개인의 에이전트 설정을 팀 전체의 일관된 기준선으로 손쉽게 전환할 수 있음을 실증합니다.

1. **컨테이너 기반의 에이전트 표준화와 신속한 배포**
   - 기존의 복잡한 마크다운과 YAML 설정 공유 방식에서 벗어나, OpenClaw를 단일 컨테이너 이미지로 패키징하여 배포 프로세스를 단순화합니다.
   - 개발자는 로컬에서 Podman 명령어를 통해 단 2초 만에 하위 에이전트(sub-agent)를 구동하고 동일한 환경을 즉시 복제할 수 있습니다.
2. **Podman과 Kubernetes를 결합한 하이브리드 스택**
   - 로컬 개발에는 Podman을 사용하고 프로덕션 배포 및 확장에는 Kubernetes를 활용하며, 전체 파이프라인에서 동일한 컨테이너 이미지를 유지합니다.
   - 데이터 백업과 복구는 볼륨(Volumes)을 통해 관리하여 에이전트의 상태와 작업 데이터를 안전하게 보존하고 지속성을 확보합니다.
3. **이중 레이어 보안을 통한 안전한 API 키 관리**
   - 호스트 레벨의 API 키는 Podman secrets를 통해 안전하게 보호하고, 컨테이너 내부에서는 OpenClaw secret refs를 사용하여 이중으로 보안을 관리합니다.
4. **Nvidia 팀의 Kubernetes 기반 모델 평가 도입 사례**
   - Nvidia의 한 팀은 10명의 엔지니어가 각각 Kubernetes에서 자체 OpenClaw를 실행하여 모델 평가(evals) 작업을 수행하는 방식으로 프로덕션에 적용했습니다.
   - 이 컨테이너화 패턴을 도입함으로써 과거에 6명의 인력이 매달려야 했던 복잡한 작업을 개별 엔지니어가 효율적으로 처리할 수 있게 되었습니다.

## [GenAI MLKit — Florina Muntenescu & Oli Gaymond, Google DeepMind](https://www.youtube.com/watch?v=owH1f0N-keY)

**한 줄 인사이트**
💡 구글은 3~4GB 크기의 Gemini Nano를 시스템 레벨(AICore)에 통합하여 개별 앱의 용량 부담을 없애고, 기기 스펙에 따라 클라우드 Gemini Flash로 자동 전환되는 하이브리드 추론을 제공합니다.

**핵심 요약**
Google DeepMind의 발표자들은 모바일 기기에서 대형 언어 모델을 효율적으로 실행하기 위한 GenAI MLKit의 아키텍처와 하이브리드 전략을 제시합니다. 이들은 3~4GB에 달하는 Gemini Nano를 개별 앱에 탑재하는 대신 시스템 레벨의 AICore를 통해 공유하고, 기기 성능 한계를 극복하기 위해 클라우드 모델과 연동하는 방식을 설명합니다. 이를 통해 개발자는 복잡한 자원 관리나 기기 파편화 걱정 없이 온디바이스 AI 기능을 구현할 수 있게 됩니다.

1. **AICore를 통한 시스템 레벨의 Gemini Nano 공유**
   - 3~4GB 크기의 Gemini Nano를 개별 앱마다 내장하는 것은 불가능하므로, 안드로이드 시스템의 AICore가 모델을 한 번만 설치해 모든 앱이 공유하도록 지원합니다.
   - 포그라운드 앱에 최우선 순위를 부여하고 백그라운드 작업은 충전 중인 야간 시간에 실행되도록 시스템이 자동 관리하여 개발자의 자원 관리 부담을 없앱니다.

2. **기기 스펙에 따른 도달 범위와 Classic MLKit의 차이**
   - GenAI MLKit API는 최근 2년 내 출시된 플래그십 기기에서만 작동하는 하드웨어 제약이 존재합니다.
   - 반면 비전 및 OCR 기능을 제공하는 기존 Classic MLKit은 전 세계 10억 대 이상의 기기에서 문제없이 실행되는 범용성을 자랑합니다.

3. **클라우드와 온디바이스를 잇는 하이브리드 추론**
   - 최근 출시된 하이브리드 추론 기능은 온디바이스 Gemini Nano를 사용할 수 없을 때 클라우드의 Gemini Flash로 자동 전환(Fallback)되도록 지원합니다.
   - 이를 통해 개발자는 기기 사양에 관계없이 사용자에게 일관된 생성형 AI 경험을 중단 없이 제공할 수 있습니다.

4. **임베딩 API 로드맵과 LiteRT 대안**
   - RAG(검색 증강 생성) 스타일의 솔루션 구축을 지원하기 위해 조만간 새로운 임베딩(Embedding) API가 출시될 예정입니다.
   - GenAI MLKit의 표준 기능 범위를 벗어나는 고도의 맞춤형 온디바이스 AI 모델 제어가 필요한 경우에는 LiteRT가 대안으로 제시됩니다.

## [Chip design from the bottom up – Reiner Pope](https://www.youtube.com/watch?v=oIk3R-sMX5o)

**한 줄 인사이트**
💡 현대 칩 디자인의 핵심 병목은 연산 속도가 아닌 데이터 이동의 물리적 비용이며, GPU와 TPU는 이 한계를 극복하기 위해 메모리 재사용을 극대화하도록 설계되었습니다.

**핵심 요약**
구글에서 TPU 아키텍처를 설계했던 MatX의 CEO 라이너 포프(Reiner Pope)는 기초적인 논리 게이트부터 시작하여 현대 AI 가속기인 TPU와 GPU가 왜 현재와 같은 구조를 갖추게 되었는지 칠판 강의를 통해 설명합니다. 그는 단순한 연산 장치에서 출발해 데이터 이동의 병목을 해결하는 시스톨릭 어레이(Systolic Array)와 파이프라인 레지스터의 역할을 규명하고, 최종적으로 CPU, GPU, 인간 뇌의 구조적 차이를 물리적 한계 관점에서 분석합니다. 이를 통해 하드웨어 설계가 소프트웨어 요구사항과 물리적 제약 사이에서 어떻게 타협점을 찾아가는지 명확한 논리적 흐름을 보여줍니다.

1. **데이터 이동의 병목과 멀티플렉서의 비용**
   - 칩 내부에서 단순 연산보다 데이터를 필요한 곳으로 이동시키는 멀티플렉서(Mux)와 배선이 차지하는 면적과 전력 소모가 훨씬 더 큽니다.
   - 데이터 이동 거리가 멀어질수록 지연 시간과 에너지 비용이 기하급수적으로 증가하기 때문에, 현대 아키텍처는 데이터 물리적 이동을 최소화하는 데 집중합니다.

2. **시스톨릭 어레이를 통한 메모리 대역폭 한계 극복**
   - 구글 TPU의 핵심인 시스톨릭 어레이(Systolic Array)는 행렬 곱셈 시 데이터를 한 번 읽어와 인접한 연산 장치들로 계속 전달하며 재사용합니다.
   - 이 구조는 메모리에서 매번 데이터를 읽어오는 병목(Memory Wall)을 해결하여, 제한된 대역폭 내에서 연산 효율을 극대화합니다.

3. **클록 사이클과 파이프라인 레지스터의 물리적 제약**
   - 신호가 칩의 물리적 거리를 이동하는 데 시간이 걸리기 때문에, 중간에 파이프라인 레지스터를 삽입하여 단계를 쪼개야 높은 클록 주파수를 유지할 수 있습니다.
   - FPGA는 프로그래밍 가능한 유연성을 제공하지만, 이 레지스터와 배선 오버헤드로 인해 전용 주문형 반도체(ASIC)보다 성능과 효율이 크게 떨어집니다.

4. **CPU, GPU, 그리고 인간 뇌의 구조적 차이**
